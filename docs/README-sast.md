# Static Application Security Testing (SAST) — Reusable Workflow

A reusable GitHub Actions workflow that runs Snyk-based security
scanning across container images, Infrastructure as Code, and
application code. Results are uploaded to GitHub's Security tab as
SARIF reports.

## Usage

Below is the most simple, working example of using the action with the
minimum required permissions.

Below is a table of each of the permissions required and why they're required:

| Name | Permission | Reason |
|---|---|---|
| `id-token` | `write` | Allows the workflow to request an OIDC token from GitHub, which is exchanged for temporary AWS credentials when assuming the ECR IAM role. |
| `contents` | `read` | Allows `actions/checkout` to clone the repository. |
| `security-events` | `write` | Allows `codeql-action/upload-sarif` to publish scan results to the GitHub Security tab. |
| `actions` | `read` | Required by `codeql-action/upload-sarif` to read workflow run metadata so it can correctly associate SARIF results with the triggering workflow. |

```yaml
jobs:
  image_scan:
    name: Run SaST tooling
    uses: ministryofjustice/laa-reusable-github-actions/.github/workflows/sast.yml@<insert latest sha here>
    permissions:
      actions: read
      id-token: write
      contents: read
      security-events: write
    secrets:
      SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
```

See [below](#Secrets) for when to use which kind of authentication.

## Inputs

The workflow provides a number of configurable inputs that adjust the
behaviour of the workflow. The workflow has been designed to require
minimal input and can have its behaviour augmented by providing
additional values, below is the listing of the current set of inputs.

Any inputs not set will just use the default value, so no need to
change anything you don't need to.

| Name | Required | Default | Description |
|---|---|---|---|
| `image_uri` | No | — | URI of the container image to scan. The workflow does not build the image; it must already exist. If omitted, the action won't do container scanning. |
| `dockerfile_path` | No | `Dockerfile` | Path to the Dockerfile used by Snyk for context during container scanning. Relative to the top-level of the repository so unless you have a monorepo this is probably fine to leave |
| `iac_path` | No | `""` | Path to Helm/IaC files. Leave unset to skip Infrastructure as Code scanning. |
| `minimum_threshold` | No | `high` | Minimum severity level to report. One of: `low`, `medium`, `high`, `critical`. |
| `ecr_session_name` | No | `laa-reusable-snyk` | Session name used when assuming the ECR role. Used for audit purposes. Unless you have a specific reason to override, you can leave this as default. |

## Secrets

The workflow also requires a couple of secrets which are confusingly
defined as not required by the pipeline.

The reason for this is we need to create a more flexible condition
than a simple `true`/`false` required option allows for regarding Snyk
authentication.

### Snyk

*Option 1: OAuth (recommended)*

The OAuth flow for Snyk is more secure than a static API key or
service account token in that it'll generate short-lived tokens, so
this would be the preferred way to manage tokens.

| Name | Description |
|---|---|
| `SNYK_CLIENT_ID` | Snyk OAuth client ID |
| `SNYK_CLIENT_SECRET` | Snyk OAuth client secret |

*Option 2: Static token*

If you would prefer using the service account, we would strongly
advise using a rotation window of no longer than **90 days** to ensure
that keys aren't left around.

| Name | Description |
|---|---|
| `SNYK_TOKEN` | Snyk API token |

### AWS

If required to pull an image from ECR, it's also possible to specify
the role to assume as a secret and the workflow will attempt to
authenticate against ECR.

This token should be generated by Cloud Platform in your secrets.

| Name | Description |
|---|---|
| `ECR_ROLE_TO_ASSUME` | IAM role ARN to assume for ECR access |

## What it scans

### Container

**Set `image_uri` to enable**

`snyk container test` will scan the image URI given and assisted by
the `Dockerfile` you specify to provide any CVEs that impact both the
application dependencies and the container itself. The container scan
capability also will run `snyk container monitor** to upload the
reports to Snyk and add the dependencies to the Snyk registry of
things to track.

### Infrastructure as Code

**Set `iac_path` to enable**

`snyk iac test` will scan against the path specified by the input to
measure against the conditions defined by the global Snyk policy. This
will change and improve over time, but is ultimately dictated by how
effective that is.

### Static Analysis

**Always enabled**

`snyk code test` will perform static analysis on the code base
regardless of any inputs you specify, so if you pass in no inputs this
is all you'll get. Depending on your use-case, that might be enoough.

## Results

SARIF output from each scan is uploaded to GitHub's "Security &
Quality" tab. Results are scoped to the repository and visible under
**Security & Quality → Code scanning alerts**.

The workflow sanitises SARIF files before upload to replace any `null`
or `undefined` severity values, which would otherwise cause the upload
to fail.

## Examples

### Scan a container image from ECR

ECR image URIs follow this format:

```
<account_id>.dkr.ecr.<region>.amazonaws.com/<repository>:<tag>
```

A typical workflow that builds and tags an image before calling this workflow:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image_uri: ${{ steps.build.outputs.image_uri }}
    steps:
      - name: Build and push image
        id: build
        env:
          REGISTRY: 123456789012.dkr.ecr.eu-west-2.amazonaws.com
          REPOSITORY: my-app
          TAG: ${{ github.sha }}
        run: |
          docker build -t $REGISTRY/$REPOSITORY:$TAG .
          docker push $REGISTRY/$REPOSITORY:$TAG
          echo "image_uri=$REGISTRY/$REPOSITORY:$TAG" >> $GITHUB_OUTPUT

  sast:
    needs: build
    uses: ministryofjustice/laa-reusable-github-actions/.github/workflows/sast.yml@<insert latest sha here>
    with:
      image_uri: ${{ needs.build.outputs.image_uri }}
    secrets:
      SNYK_CLIENT_ID: ${{ secrets.SNYK_CLIENT_ID }}
      SNYK_CLIENT_SECRET: ${{ secrets.SNYK_CLIENT_SECRET }}
      ECR_ROLE_TO_ASSUME: ${{ secrets.ECR_ROLE_TO_ASSUME }}
```

### Scan IaC alongside a container image

```yaml
  sast:
    uses: ministryofjustice/laa-reusable-github-actions/.github/workflows/sast.yml@<insert latest sha here>
    with:
      image_uri: ${{ needs.build.outputs.image_uri }}
      iac_path: helm/deployment
      minimum_threshold: medium
    secrets:
      SNYK_CLIENT_ID: ${{ secrets.SNYK_CLIENT_ID }}
      SNYK_CLIENT_SECRET: ${{ secrets.SNYK_CLIENT_SECRET }}
      ECR_ROLE_TO_ASSUME: ${{ secrets.ECR_ROLE_TO_ASSUME }}
```
